- **Verified against the repo — this is a config-on-an-existing-base slice with ZERO new shared code**: Confirmed directly. The reusable base already exists (`core/crates/kaname-core/src/statement/ledger_reader.rs`): its `LedgerReaderConfig` trait recognizes a transaction against an **ordered list** of anchor patterns (`anchor_res() -> Vec<&Regex>`, "tried in order (first match wins)", `ledger_reader.rs:60`) — so Federal's two templates are pure configuration (proven by HDFC's two layouts); its `anchor_amount`/`loose_amount` already resolve a two-column withdrawal/deposit pair and parse **bare-integer** amounts via `Decimal::from_str_exact` (`ledger_reader.rs:238-268`) — so the Fi whole-number amounts (`5000`/`50000`) need no new capability; its `find_anchors` captures a `serial` named group **separately** from `desc` (`ledger_reader.rs:220-221`) — so the S-serial is captured out of the description for free; and its stitcher skips only `is_balance_line` matches (`ledger_reader.rs:302`, `395-400`) — so the `GRAND TOTAL` line (which matches no anchor and no opening/closing pattern) is folded into the last row's narration exactly as the ground truth shows.
- **Verified — the shared account-tail helper already exists (no new shared helper)**: `core/crates/kaname-core/src/statement/common.rs:177-188` already provides `account_tail_last4(text, primary)` (per-bank primary regex → trailing four, else the longest standalone ≥9-digit run → trailing four), used by both `icici_bank.rs` and `hdfc_bank.rs`. Federal reuses it with its own primary regex (`Account Number` + optional `X*` mask + 4+ digits), matching both the classic full number (`99990100001234`→`1234`) and the Fi mask (`XXXXX4222`→`4222`). This is the key difference from the HDFC slice: HDFC had to *factor out* the helper; Federal adds **no** new shared code at all (FR-024, US9, SC-010).
- **Verified — the shared date parser already carries all three Federal formats**: `common.rs:24-36` already lists `%d/%m/%Y` (Fi), `%Y-%m-%d` (classic ISO period), and `%d-%b-%Y` (classic `DD-MON-YYYY`, annotated "HDFC" but format-identical), so no date-parser change is needed — supporting FR-026 and the "no new shared engine internals / no new date format" boundary.
- **Verified — the shared `FEDERAL` bank code and the coexistence design**: `core/crates/kaname-core/src/statement/federal.rs:21` sets `BANK_CODE = "FEDERAL"` for the already-landed Scapia/Federal **credit-card** reader (claim markers `["Scapia","Federal Bank"]`), and `docs/kaname-ios-plan.md:51` lists `federal_bank` as a distinct bank-account reader under a registry keyed by `(bank_code, account_kind)`.
